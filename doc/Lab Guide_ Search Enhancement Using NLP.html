Lab 1: Overview

Objective: In this lab, we are going to see how we can use Elasticsearch and NLP to improve your search. We are going to index documents with and without enrichment and compare the performance of the searches on the two different kind of documents.

  1. Before starting let’s add data to our Elasticsearch cluster! First of all ssh into server1.

    ssh server1

    Now we can extract Elasticsearch and Kibana:

    tar -zxf elasticsearch-6.4.1.tar.gz
tar -zxf kibana-6.4.1-linux-x86_64.tar.gz

    Start Elasticsearch in this tab using the following command:

    ./elasticsearch-6.4.1/bin/elasticsearch -E network.host=0.0.0.0

    Open a new terminal tab and start kibana:

    ssh server1
./kibana-6.4.1-linux-x86_64/bin/kibana --host=0.0.0.0

    You can now, go into Kibana, click here

    Go into the "Dev tool":

    Let’s start by indexing some documents:

    POST test/_doc/_bulk
{"index": {"_id":1}}
{"message": "He became rich by selling his business."}
{"index": {"_id":2}}
{"message": "Rich visited the beautiful city of Rouen passing just in front of the cathedrale."}
{"index": {"_id":3}}
{"message": "Rich?? I don't think I am. I would even say that I'm poor."}
{"index": {"_id":4}}
{"message": "Rich and wealthy share similar meaning but may have different usage."}
{"index": {"_id":5}}
{"message": "One law for the rich another for the poor."}

    Now let’s say that you want to retrieve the document speaking about Rich (the person). How can you do that? You could try to do a search query that will match "rich".

    GET test/_search
{
  "query": {
    "match": {
      "message": "rich"
    }
  }
}

    The document containing "Rich" the person has not scored well. It is actually the last one.

  2. To improve the position of the document we could try to change the analyzer because after all, a first name always start with a upper case. Let’s create an index2 for documents that will have a single field message analyzed using a custom analyzer with only a standard tokenizer, so no lowercasing.

    PUT test2
{
  "settings": {
    "analysis": {
      "analyzer": {
        "simple_tokenizer": {
          "type": "custom",
          "tokenizer": "standard"
        }
      }
    }
  },
  "mappings": {
    "_doc":{
      "properties":{
        "message": {
          "type": "text",
          "analyzer": "simple_tokenizer"
        }
      }
    }
  }
}

  3. Let’s reindex the documents in our test index into our test2 index using the reindex API.

    POST _reindex
{
  "dest": {
    "index": "test2"
  },
  "source": {
    "index": "test"
  }
}

  4. Now that a new analysis has been applied to the documents let’s try to search for "Rich" with an upper case this time.

    GET test2/_search
{
  "query": {
    "match": {
      "message": "Rich"
    }
  }
}

    This time less documents are returned, but still, the one that are interesting for the problem are ranked last. How can it be improved?

  5. NLP can solve this problem. What if the documents we have indexed previously had more information? Let’s reindex documents that contain Part Of Speech (POS) tag.

    POST test3/_doc/_bulk
{"index": {"_id":1}}
{"message": "He became rich by selling his business","enriched_message": "he_PRP became_VBD rich_JJ by_IN selling_VBG his_PRP$ business_NN"}
{"index": {"_id":2}}
{"message": "Rich visited the beautiful city of Rouen passing just in front of the cathedrale.","enriched_message": "rich_NNP visited_VBD the_DT beautiful_JJ city_NN of_IN Rouen_NNP passing_VBG just_RB in_IN front_NN of_IN the_DT cathedrale_NN."}
{"index": {"_id":3}}
{"message": "Rich?? I don't think I am. I would even say that I'm poor.","enriched_message": "rich_JJ?? i_PRP do_VBP n't_RB think_VB i_PRP am_VBP. i_PRP would_MD even_RB say_VB that_IN I_PRP 'm_VBP poor_JJ."}
{"index": {"_id":4}}
{"message": "Rich and wealthy share similar meaning but may have different usage.","enriched_message": "rich_JJ and_CC wealthy_JJ share_VB similar_JJ meaning_NN but_CC may_MD have_VB different_JJ usage_NN."}
{"index": {"_id":5}}
{"message": "One law for the rich another for the poor.","enriched_message": "one_CD law_NN for_IN the_DT rich_JJ another_DT for_IN the_DT poor_JJ."}

  6. Now we can search for "Rich" as a proper noun. Do a search query to target the proper noun Rich (the tag for a proper noun is NNP).

    GET test3/_search
{
  "query": {
    "match": {
      "enriched_message": "rich_NNP"
    }
  }
}

  7. This search give us the expected result but it give us only one document. Do a query that will give us the expected document ranked number one and all the other documents that contain Rich but not as a pronoun.

    GET test3/_search
{
  "query": {
    "bool": {
      "must": [
        {
          "match": {
            "message": "rich"
          }
        }
      ],
      "should": [
        {
          "match": {
            "enriched_message": "rich_NNP"
          }
        }
      ]
    }
  }
}

  8. That was one example where NLP can help improving search. Let’s see another example, this time using entity recognition. The following document contains multiple entity:

    {
  "message" : "Kobe Bryant was one of the best basketball players of all times. Not even Michael Jordan has ever scored 81 points in one game. Munich is really an awesome city, but New York is as well. Yesterday has been the hottest day of the year."
}

    It contains the following: "Kobe Bryant", "Michael Jordan", "Munich", "New York" , "Yersteday". Some informations are dates, some are names and others are locations. Being able to automatically detect and extract those informations can be very powerful.

  9. Let’s index some documents that don’t have enrichment.

    POST test4/_doc/_bulk
{"index": {"_id":1}}
{"message": "Nathan is living in the busy city of New York"}
{"index": {"_id":2}}
{"message": "Bill is going to visit Marc in the city of New York"}
{"index": {"_id":3}}
{"message": "Bill doesn't really like the city of New York and prefer the calm state of Missouri"}
{"index": {"_id":4}}
{"message": "The catacombes are like a small city under Paris, this is Melvyn's favorite place."}

    How can we build a query that run a term aggregation on the documents that contain "New York"? First build a query that target the documents containing "New York".

    GET test4/_search
{
  "query": {
    "match": {
      "message": "new york"
    }
  }
}

    3 documents mention New York. Now if you want to do the same thing for every locations in your data, it will be hard. A query will need to be be done for every single location.

  10. Instead of iterating on every locations present in our documents, we could process those documents and enrich them. Let’s do it:

    POST test5/_doc/_bulk
{"index": {"_id":1}}
{"message": "Nathan is living in the busy city of New York", "entities" : {"locations" : ["New York"], "names" : [ "Nathan"]}}
{"index": {"_id":2}}
{"message": "Bill is going to visit Marc in the city of New York", "entities" : {"locations" : ["New York"], "names" : [ "Bill", "Marc"]}}
{"index": {"_id":3}}
{"message": "Bill doesn't really like the city of New York and prefer the calm state of Missouri", "entities" : {"locations" : ["New York", "Missouri"], "names" : [ "Bill"]}}
{"index": {"_id":4}}
{"message": "The catacombes are like a small city under Paris, this is Melvyn's favorite place.", "entities" : {"locations" : ["Paris"], "names" : ["Melvyn"]}}

  11. Instead of doing a query, a term aggregation can be done on the field entities.names.keyword.

    GET test5/_search
{
  "aggs": {
    "entities_name": {
      "terms": {
        "field": "entities.locations.keyword",
        "size": 10
      }
    }
  }
}

    There is 3 documents about New York, 1 about Paris and 1 about Missouri.

    Summary: It’s over! In this unit you saw how NLP can help in many ways. Enriching your documents will improve the quality of your searches, having more structure in your data on unstructured fields (text fields) can be powerful and improve your analytic. In the next unit you are going to learn how to use a NLP solution to enrich your documents.

End of Lab 1

Lab 2: NLP

Objective: In this lab, we are going to see how we can use enriched documents to enhance the search experience. NLTK is going to be uses in order to do the enrichment.

  1. In order to enrich our documents we are going to need an application that do POS tagging or entity detection. Hopefully one is already built. Let’s use it. In the nlp_api folder there is a requirements.txt file. Run the following:

    virtualenv -p python3 venv
source venv/bin/activate
pip install -r requirements.txt

  2. Now that we have our virtual environment set up with all the dependencies, let’s download some required ressources form nltk:

    python
import nltk
nltk.download('maxent_ne_chunker')
nltk.download('words')
nltk.download('punkt')
nltk.download('averaged_perceptron_tagger')
exit()

    Start the API that will enrich the documents:

    python nlp_api.py

    The API should be up and running. Let’s work with the following data:

    {
  "original": "John is rich, he just won the lottery."
}

{
  "original": "Rich is an Elasticsearch certified engineer."
}

  3. Index those two documents.

    PUT no_nlp/_doc/1
{
  "original": "John is rich, he just won the lottery."
}

PUT no_nlp/_doc/2
{
  "original": "Rich is an Elasticsearch certified engineer."
}

  4. Do a search for "rich" in the index no_nlp.

    GET no_nlp/_search
{
  "query": {
    "match": {
      "original": "rich"
    }
  }
}

  5. What is the POS tag of "rich" in the first and second document?

    In the first document rich is an adjective: JJ In the second document rich is a noun: NN and even a first name so: NNP

  6. Let’s see if our API can detect it. Using curl send a POST request on: http://localhost:5000/pos_tag. Pass along the document that you want to analyze into the body using the submit key.

    curl -d '{"submit":"John is rich, he just won the lottery."}' -H "Content-Type: application/json" -X POST http://localhost:5000/pos_tag
curl -d '{"submit":"Rich is an Elasticsearch certified engineer."}' -H "Content-Type: application/json" -X POST http://localhost:5000/pos_tag

  7. Index the two resulting documents in the nlp_pos_tag index.

    PUT nlp_pos_tag/_doc/1
{
  "enriched": "John_NNP is_VBZ rich_JJ ,_, he_PRP just_RB won_VBD the_DT lottery_NN ._.",
  "original": "John is rich, he just won the lottery."
}

PUT nlp_pos_tag/_doc/2
{
  "enriched": "Rich_NNP is_VBZ an_DT Elasticsearch_NNP certified_VBD engineer_NN ._.",
  "original": "Rich is an Elasticsearch certified engineer."
}

  8. Search for rich as a noun.

    GET nlp_pos_tag/_search
{
  "query": {
    "match": {
      "enriched": "rich_nnp"
    }
  }
}

  9. We only did POS tagging so far. Let’s try to do some named entity recognition. In the API that you start earlier there is another endpoint available: http://localhost:5000/entity. Let’s use this endpoint on the following texts:

    Tom and Jerry are fighting over a piece of cheese worth 5$.
Elastic founded in 2012 in Amsterdam is behind Elasticsearch.
Jerry and Bob are going to join Elastic.
    curl -d '{"submit":"Jerry is working for the company since a long time."}' -H "Content-Type: application/json" -X POST http://localhost:5000/entity
curl -d '{"submit":"Tom and Jerry are fighting over a piece of cheese worth 5$."}' -H "Content-Type: application/json" -X POST http://localhost:5000/entity
curl -d '{"submit":"Jerry and Bob are going to join the company in London"}' -H "Content-Type: application/json" -X POST http://localhost:5000/entity

  10. Index the resulting documents inside the index nlp_entity.

    PUT nlp_entity/_doc/1
{
  "entities": {
    "PERSON": [
      "Jerry"
    ]
  },
  "original": "Jerry is working for the company since a long time."
}

PUT nlp_entity/_doc/2
{
  "entities": {
    "PERSON": [
      "Tom",
      "Jerry"
    ]
  },
  "original": "Tom and Jerry are fighting over a piece of cheese worth 5$."
}

PUT nlp_entity/_doc/3
{
  "entities": {
    "GPE": [
      "London"
    ],
    "PERSON": [
      "Jerry",
      "Bob"
    ]
  },
  "original": "Jerry and Bob are going to join the company in London"
}

  11. The texts were previously unstructured. More structure has been added to the documents. What is the most frequent name in our documents?

    GET nlp_entity/_search
{
  "size": 0,
  "aggs": {
    "name_count": {
      "terms": {
        "field": "entities.PERSON.keyword"
      }
    }
  }
}

Summary: Well done. You just saw how we can use an external API to enrich documents using POS tagging and named entity recognition. Some information are lost during the indexing because of the automatic mapping. Because the enrichment is done using an external API we had to define the tokenizer in the application, we need to reflect that in our Elasticsearch mapping accordingly.

End of Lab 2

Lab 3: Ingest Opennlp

Objective: Ingest pipelines allow the processing of documents before indexation in Elasticsearch. In this lab we are going to use the Opennlp ingest pipeline in order to enrich documents with NLP information. This pipeline will do named entity extraction and add new fields to the indexed documents. This lab is going to guide you along on how you can install, use and configure your pipeline.

  1. Before installing a plugin make sure to stop your Elasticsearch node.

  2. Now that Elasticsearch is not running anymore you should install the ingest pipeline plugin. To do that use the Elasticsearch script plugin manager.

    elasticsearch-6.4.1/bin/elasticsearch-plugin install https://github.com/spinscale/elasticsearch-ingest-opennlp/releases/download/6.4.1.1/ingest-opennlp-6.4.1.1.zip

    It should take few seconds to be installed.

  3. The plugin come with models to do named entity extraction. You need to specify which model you want to use in the elasticsearch.yml file. Update the file to add those parameters then restart Elasticsearch:

    ingest.opennlp.model.file.persons: en-ner-persons.bin
ingest.opennlp.model.file.dates: en-ner-dates.bin
ingest.opennlp.model.file.locations: en-ner-locations.bin
    vim config/elasticsearch.yml

    Copy paste the above parameters and restart Elasticsearch.

  4. If we configure an ingest pipeline, it will now have the possibility of detecting persons, dates and locations. Let’s configure an ingest pipeline called per_dat_loc_extraction that do the above. Give him the following description: "A pipeline to do dates, locations and persons extraction". The pipeline must process the original field.

    PUT _ingest/pipeline/per_dat_loc_extraction
{
  "description": "A pipeline to do dates, locations and persons extraction",
  "processors": [
    {
      "opennlp" : {
        "field" : "original"
      }
    }
  ]
}

  5. You just created an ingest pipeline that do named entity extraction. Let’s use it!

    {
  "original" : "John Smith has been working in New York since September 2015"
}

{
  "original" : "Paris is the capitol of France."
}

{
  "original" : "John Smith founded the company in 2012"
}

  6. Index the above documents in an index called ingest_pipeline_index using the pipeline we defined perviously.

    PUT ingest_pipeline_index/_doc/1?pipeline=per_dat_loc_extraction
{
  "original" : "John Smith has been working in New York since September 2015"
}

PUT ingest_pipeline_index/_doc/2?pipeline=per_dat_loc_extraction
{
  "original" : "Paris is the capitol of France."
}

PUT ingest_pipeline_index/_doc/3?pipeline=per_dat_loc_extraction
{
  "original" : "John Smith founded the company in 2012"
}

  7. Retrieve the document you just indexed and look at their new fields.

    GET ingest_pipeline_index/_search

    The first document has three new fields: entities.persons, entities.locations and entities.dates. The second document has one new field: entities.locations. The third documents has two new fields: entities.persons and entities.dates.

  8. Let’s define another ingest pipeline but this time, your pipeline will only detect locations. Call this new pipeline: loc_extraction and give it the following description: "A pipeline to do locations extraction"

    PUT _ingest/pipeline/loc_extraction
{
  "description": "A pipeline to do locations extraction",
  "processors": [
    {
      "opennlp" : {
        "field" : "original",
        "fields" : [ "locations" ]
      }
    }
  ]
}

  9. Reindex the previous documents using loc_extraction pipeline.

    PUT ingest_pipeline_index/_doc/1?pipeline=loc_extraction
{
  "original" : "John Smith has been working in New York since September 2015"
}

PUT ingest_pipeline_index/_doc/2?pipeline=loc_extraction
{
  "original" : "Paris is the capitol of France."
}

PUT ingest_pipeline_index/_doc/3?pipeline=loc_extraction
{
  "original" : "John Smith founded the company in 2012"
}

    You can have a look at the reindexed documents by using the _search endpoint.

  10. Now what is happening if you index the following document using per_dat_loc_extraction pipeline:

    {
  "original" : "Goldman Sachs was founded in 1869"
}
    PUT ingest_pipeline_index/_doc/4?pipeline=per_dat_loc_extraction
{
  "original" : "Goldman Sachs was founded in 1869"
}

GET ingest_pipeline_index/_doc/4

    The fact that a company name is present and not detected can be an issue. There is only one new field that has been created: entities.dates

  11. Let’s add a model that can detect organizations. In the folder datasets there is a file that contains a model to detect organizations. Move this file in the folder config/ingest-opennlp in the elasticsearch repository.

    mv datasets/en-ner-organization.bin elasticsearch-6.4.1/config/ingest-opennlp

    Now stop your Elasticsearch node and update your configuration file elasticsearch.yml and add the following:

    ingest.opennlp.model.file.organizations: en-ner-organization.bin

    Create an ingest pipeline called per_dat_loc_org_extraction

    PUT _ingest/pipeline/per_dat_loc_org_extraction
{
  "description": "A pipeline to do dates, locations, persons and organizations extraction",
  "processors": [
    {
      "opennlp" : {
        "field" : "original"
      }
    }
  ]
}

  12. Index the document using the new pipeline. What changed?

    PUT ingest_pipeline_index/_doc/4?pipeline=per_dat_loc_extraction
{
  "original" : "Goldman Sachs was founded in 1869"
}

GET ingest_pipeline_index/_doc/4

    This will output the following:

    {
  "_index": "ingest_pipeline_index",
  "_type": "_doc",
  "_id": "4",
  "_version": 2,
  "found": true,
  "_source": {
    "original": "Goldman Sachs was founded in 2000",
    "entities": {
      "organizations": [
        "Goldman Sachs"
      ],
      "dates": [
        "2000"
      ]
    }
  }
}

    A new organizations field is present and contains the detected organization. "Goldman Sachs".

Summary: This is the end of the third lab! Congratulation! One more lab to go. Now that we can enrich our documents using ingest pipeline directly inside Elasticsearch, let’s see how we can leverage those enriched documents to improve our searches and do visualizations. We are going to see as well what are the limit and improvements to this system that can be done. === End of Lab 3 '''

Lab 4: Enhanced Search

Objective: In the previous labs you learned how to use a client to enrich your document using NLP (POS tagging and named entity recognition), then we saw how to use opennlp ingest pipeline to do named entity recognition directly on our documents before indexation. In this lab we are going to see some tips in order to leverage those information in our searches.

  1. We are going to index a bunch of documents. Let’s first have a look at it! In the datasets folder there is a file called: sample.txt.

    cut -c-500 sample.txt

  2. We want to apply NLP on those documents. So let’s define an ingest pipeline called: lab4_ingest_pipeline. This pipeline will do person and location detection.

    PUT _ingest/pipeline/lab4_ingest_pipeline
{
  "description": "A pipeline to do locations and persons recognition",
  "processors": [
    {
      "opennlp" : {
        "field" : "original",
        "fields" : [ "locations", "persons" ]
      }
    }
  ]
}

  3. Launch the scripts: scripts/lab4_index.py, it will index the document that you previously looked at. Make sure that you are in the virtual env before launching the script. You should see something before the command line in the terminal: (venv)

    python scripts/lab4_index.py

  4. Using the ingest pipeline our documents went from an unstructured form to a more structured form. We can view it using a visualization inside Kibana. Create an index pattern, go into: Visualization, Create a new visualization, Vertical bar chart, display on the Y-axis the most frequent locations using a terms aggregation.

  5. Now if the NLP for our documents is done on the client side that means that you defined your analyzer (tokenizer included) on the client side. You should take this into consideration while building the mapping for those documents in order to avoid a double analysis (and tokenization). Let’s say you have the following document that is sent by your client.

    {
  "enriched": "John_NNP is_VBZ rich_JJ ,_, he_PRP just_RB won_VBD the_DT lottery_NN ._.",
  "original": "John is rich, he just won the lottery."
}

    You ideally don’t want to reanalyzed and retokenized the enriched field. Build a mapping for an index called e_commerce and a document type _doc that will preserve the token provided in the enriched field and index them as they are.

    PUT e_commerce
{
  "mappings": {
    "_doc": {
      "properties": {
        "original": {
          "type": "text"
        },
        "enriched": {
          "type": "text",
          "analyzer": "whitespace"
        }
      }
    }
  }
}

  6. Now let’s index some documents on which we applied some NLP that detect relationships between words inside the index you just created.

    {
  "original": "There is a discount on all black shoes",
  "enriched": "There is a discount on all black_shoes shoes"
}

{
  "original": "Huge discount on black T-shirts and white shoes",
  "enriched": "Huge_discount discount on black_T-shirts T-shirts and white_shoes shoes"
}

{
  "original": "I sell those nice white shoes",
  "enriched": "I sell those nice_shoes white_shoes shoes"
}
    PUT e_commerce/_doc/1
{
  "original": "There is a discount on all black shoes",
  "enriched": "There is a discount on all black_shoes shoes"
}

PUT e_commerce/_doc/2
{
  "original": "Huge discount on black T-shirts and white shoes",
  "enriched": "Huge_discount discount on black_T-shirts T-shirts and white_shoes shoes"
}

PUT e_commerce/_doc/3
{
  "original": "I sell those nice white shoes",
  "enriched": "I sell those nice_shoes white_shoes shoes"
}

  7. Let’s say that one of the user is trying to do the following search: black shoes. Do a search to retrieve the most relevant documents.

    Having issues? If you are doing a search on the original field then you will retrieve all the documents. If you are doing a search on the enriched field, then you are going to match all the documents as well. The problem is that no NLP has been applied on the search request. Now if we apply NLP on the search request this is what the request will probably look like: black_shoes shoes. Now do a search using this processed search.

    GET e_commerce/_doc/_search
{
  "query": {
    "match": {
      "enriched": "black_shoes shoes"
    }
  }
}

Summary: This is the end of the 4th lab. You should have enough knowledge to implement such a solution inside your own cluster. Don’t forget that the best solutions rely most of the time on your own datasets.

End of Lab 4
